# Figma MCP is the canonical design input; HTML/ASCII fallback only

## Context and Problem Statement

`grimoire-design` produces design artifacts and `grimoire-draft` consumes them to generate Gherkin tasks. What format should designs use?

Options range from a custom grimoire DSL (YAML/JSON of layout primitives) to existing tool-native formats (Figma file references, exported PNGs, HTML).

## Decision Drivers

- User explicitly rejected inventing a custom design DSL — "I don't want to invent my own design tool language in YAML/PNG"
- Must work with existing toolchains designers already use
- Must be verifiable downstream (Gherkin generation needs to parse component states)
- Must degrade gracefully when no design tool is available
- Tests/CI should be able to consume the same format

## Considered Options

1. **Figma MCP primary, HTML/ASCII fallback** — Figma file URL + MCP server in config; if absent, default to HTML preview generated by grimoire-design; ASCII for simple cases
2. **Custom grimoire design YAML** — invent a `design.yml` schema describing layout, components, states
3. **PNG/SVG exports** — designer attaches static images; AI reads via vision
4. **Storybook stories** — every design is a `.stories.tsx` file; reuse Storybook MCP

## Decision Outcome

Chosen option: **Option 1**.

**Primary:** Figma MCP (or generic design-tool MCP set up at onboarding) provides frame/variable/component data. Designs are referenced by Figma URL + node ID in `manifest.md` and `.grimoire/changes/<id>/designs/`.

**Fallback chain:**
- If MCP unavailable AND change is non-trivial → HTML preview generated by `grimoire-design`, stored at `.grimoire/changes/<id>/designs/preview.html`
- If MCP unavailable AND change is trivial (e.g., copy change, simple form) → ASCII wireframe in markdown

**Not adopted:** custom YAML schema. PNG/SVG only as supplementary asset (e.g., logos in `.grimoire/brand/`), not as the design definition.

### Consequences

- Good: Zero invented format — designers and tools already understand Figma + HTML
- Good: Figma MCP gives AI structured access to tokens, components, states
- Good: HTML fallback is testable (axe-core runs on it, Playwright can render it)
- Good: Visual fidelity tier (token-lint + axe-core) can operate on the same HTML preview that the designer sees
- Bad: Figma MCP is read-only — can't push generated frames back to Figma (v1 limit)
- Bad: HTML preview needs basic component skeleton; risk of generating "generic SaaS look" without brand-tokens grounding
- Bad: Non-Figma designers (Sketch, Penpot, Framer) get generic MCP setup with stub instructions rather than first-class integration

### Confirmation

If a designer can open a Figma file URL in `manifest.md`, run `grimoire-draft`, and have generated Gherkin scenarios that correctly reference component states visible in Figma, the decision is validated.
